# Deployment Policy

This project is deployed on **Vercel** as a Vite + React SPA backed by **Supabase**. Follow these rules for every deployment.

## Rules (MUST FOLLOW)

- **Never deploy directly from `main` without passing CI checks**
  - All merges to `main` must pass lint, build, and any automated tests.
  - A broken build on `main` must be treated as a P1 incident.

- **Environment variables must be set in Vercel — never hard-coded**
  - All secrets (`VITE_SUPABASE_ANON_KEY`, etc.) live in Vercel's environment variable settings.
  - Verify the correct variables are set for each environment (Production, Preview, Development).
  - See [ENV_VARIABLES.md](./ENV_VARIABLES.md) for the full variable reference.

- **The `dist/` folder is never committed**
  - Build output is generated by Vercel's build pipeline.
  - `dist/` is gitignored — keep it that way.

- **Preview deployments are not production**
  - Preview deployments (triggered by PRs) should use a **staging Supabase project**, not production.
  - Never point a preview deployment at the production database.

- **Production deploys require a working build**
  - Run `npm run build` locally to confirm the build is clean before merging.
  - Fix all TypeScript/Vite errors before shipping.

## Deployment Environments

| Environment | Trigger | Supabase Project | URL |
|---|---|---|---|
| **Production** | Merge to `main` | Production project | `your-domain.vercel.app` |
| **Preview** | Pull request / branch push | Staging project | Auto-generated Vercel URL |
| **Local dev** | `npm run dev` | Local / staging project | `localhost:5173` |

## Vercel Configuration

The project includes a [`vercel.json`](./vercel.json) that configures routing for the SPA.

- **SPA fallback**: All routes must fall through to `index.html` so that React Router handles them client-side.
- **Headers**: Add security headers in `vercel.json` where possible:

  ```json
  {
    "headers": [
      {
        "source": "/(.*)",
        "headers": [
          { "key": "X-Content-Type-Options", "value": "nosniff" },
          { "key": "X-Frame-Options", "value": "DENY" },
          { "key": "Referrer-Policy", "value": "strict-origin-when-cross-origin" },
          { "key": "Permissions-Policy", "value": "camera=(), microphone=(), geolocation=()" }
        ]
      }
    ]
  }
  ```

## Pre-Deployment Checklist

Before every production deployment:

- [ ] `npm run build` completes with no errors locally
- [ ] All required environment variables are set in Vercel (Production environment)
- [ ] No secrets or credentials are hard-coded in source files
- [ ] RLS policies and Supabase migrations are applied to the production project
- [ ] `vercel.json` SPA routing is correct
- [ ] Preview deployment tested and approved on staging data
- [ ] `SECURITY.md` pre-merge checklist passed

## Rolling Back

If a production deployment causes issues:

1. Go to the Vercel dashboard → **Deployments**.
2. Find the last known-good deployment.
3. Click **Promote to Production**.
4. Investigate the root cause before re-deploying.
5. For database issues, restore from the Supabase backup taken before the migration.
